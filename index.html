Chapter 3: End-to-End ML Workflow

Prerequisites: None. This chapter provides context for every subsequent technique.
Learning Objectives
  • Describe the complete ML lifecycle from problem definition to monitoring
  • Identify where each algorithm and technique fits in the workflow
  • Apply professional workflow practices to your own projects

Every machine learning project follows a lifecycle. Understanding this workflow before learning individual algorithms prevents the common mistake of jumping to complex models before understanding the problem or validating the data.

The Professional ML Lifecycle

Stage 1: Business or Research Problem Definition

Before writing any code, answer these questions:

  • What decision will the model support?
  • What is the unit of prediction (customer, transaction, image, time window)?
  • When does the target variable become known relative to prediction time?
  • What inputs are legitimately available at inference time?
  • What error is most harmful (false positives vs. false negatives)?
  • What baseline must the model beat?

Stage 2: Data Acquisition and Documentation

Collect data sources and create a data dictionary documenting each field's meaning, type, source, and update frequency. Establish data lineage: where does each feature come from, and when does it become available?

Stage 3: Data Validation and Leakage Assessment

Check for data quality issues: missing values, duplicates, outliers, schema violations. Most critically, assess target leakage: are any features created using information that would not be available at prediction time?

Critical: Target leakage is one of the most common failure modes in production ML. A model with leakage will show excellent validation metrics and fail completely in deployment. See Chapter 20 for a comprehensive leakage taxonomy.

Stage 4: Train/Validation/Test Design

Split data based on the problem structure:

  • Random split: For IID data with no temporal or group structure
  • Time-based split: For temporal data; training period strictly before validation period
  • Group-based split: For grouped data (patients, users); same group cannot appear in multiple splits
  • Stratified split: When class labels are imbalanced; preserve label distribution across splits

Stage 5: Baseline Model

Establish a simple, fast baseline before trying complex models:

  • Classification: majority-class classifier, logistic regression
  • Regression: mean/median predictor, linear regression
  • Ranking: popularity-based, random
Rule: A complex model is only justified when it reliably beats a simple baseline by a margin that justifies its added complexity, training cost, and maintenance burden.

Stage 6: Feature and Model Experiments

Experiment systematically:

  1. Start with domain-relevant features
  2. Try 2-3 model families appropriate for the data size and type
  3. Use cross-validation for model selection, never the test set
  4. Track all experiments with version-controlled parameters and results

Stage 7: Evaluation and Error Analysis

Evaluate on the held-out test set once, then perform error analysis:

  • Which subgroups does the model perform poorly on?
  • What do failure cases have in common?
  • Are errors random or systematic?
  • Is the model calibrated? (predicted probabilities match true frequencies)

Stage 8: Deployment

Deploy the model as a service with:

  • Input validation and schema checks
  • Health checks and logging
  • Versioned model artifacts
  • Rollback capability

Stage 9: Monitoring, Retraining, and Governance

Production ML models degrade over time due to data drift, concept drift, and changing environments. Monitor:

  • Service metrics: latency, throughput, error rates
  • Data quality: missing rates, distribution shifts
  • Model performance: accuracy decay, calibration drift
  • Fairness: performance across demographic groups

4.5 Implementing Statistics from Scratch

Before using NumPy, understand how these functions work:

def mean(values):
    """Compute arithmetic mean."""
    return sum(values) / len(values)

def variance(values, sample=True):
    """Compute variance. Set sample=False for population variance."""
    m = mean(values)
    squared_diffs = [(x - m) ** 2 for x in values]
    n = len(values) - 1 if sample else len(values)
    return sum(squared_diffs) / n

def std_dev(values, sample=True):
    """Compute standard deviation."""
    return variance(values, sample) ** 0.5

def median(values):
    """Compute median."""
    sorted_vals = sorted(values)
    n = len(sorted_vals)
    mid = n // 2
    if n % 2 == 0:
        return (sorted_vals[mid - 1] + sorted_vals[mid]) / 2
    return sorted_vals[mid]

def normalize(values):
    """Min-max normalize to [0, 1]."""
    min_val = min(values)
    max_val = max(values)
    rng = max_val - min_val
    if rng == 0:
        return [0.0] * len(values)
    return [(x - min_val) / rng for x in values]

def standardize(values):
    """Z-score standardization."""
    m = mean(values)
    s = std_dev(values)
    if s == 0:
        return [0.0] * len(values)
    return [(x - m) / s for x in values]

4.6 Classification Metrics from Scratch

def accuracy(y_true, y_pred):
    correct = sum(yt == yp for yt, yp in zip(y_true, y_pred))
    return correct / len(y_true)

def precision_recall_f1(y_true, y_pred, pos_label=1):
    tp = sum(1 for yt, yp in zip(y_true, y_pred)
             if yt == pos_label and yp == pos_label)
    fp = sum(1 for yt, yp in zip(y_true, y_pred)
             if yt != pos_label and yp == pos_label)
    fn = sum(1 for yt, yp in zip(y_true, y_pred)
             if yt == pos_label and yp != pos_label)

    prec = tp / (tp + fp) if (tp + fp) > 0 else 0.0
    rec = tp / (tp + fn) if (tp + fn) > 0 else 0.0
    f1 = 2 * prec * rec / (prec + rec) if (prec + rec) > 0 else 0.0
    return {"precision": prec, "recall": rec, "f1": f1}

def confusion_matrix(y_true, y_pred, labels=None):
    """Return confusion matrix as nested dict."""
    if labels is None:
        labels = sorted(set(y_true) | set(y_pred))
    matrix = {t: {p: 0 for p in labels} for t in labels}
    for yt, yp in zip(y_true, y_pred):
        matrix[yt][yp] += 1
    return matrix

5.2 Building a Dataset Summary Class

class DatasetSummary:
    """Encapsulates dataset statistics and validation."""

    def __init__(self, name: str, features: list, labels: list):
        self.name = name
        self.features = features  # List of feature vectors
        self.labels = labels
        self.n_samples = len(features)
        self.n_features = len(features[0]) if features else 0
        self._validated = False

    def validate(self) -> bool:
        """Check consistency of dataset."""
        assert len(self.features) == len(self.labels), \
            "Features and labels must have same length"
        assert self.n_samples > 0, "Dataset must not be empty"
        feature_lengths = [len(f) for f in self.features]
        assert len(set(feature_lengths)) == 1, \
            "All feature vectors must have same length"
        self._validated = True
        return True

    def class_distribution(self) -> dict:
        """Return count per class label."""
        from collections import Counter
        return dict(Counter(self.labels))

    def __repr__(self):
        status = "validated" if self._validated else "unvalidated"
        return (f"DatasetSummary(name='{self.name}', "
                f"samples={self.n_samples}, "
                f"features={self.n_features}, "
                f"classes={len(set(self.labels))}, "
                f"{status})")

5.3 Building a StandardScaler from Scratch

class StandardScaler:
    """Standardize features by removing mean and scaling to unit variance.

    Follows scikit-learn's fit/transform pattern.
    """

    def __init__(self):
        self.means = None
        self.stds = None
        self._fitted = False

    def fit(self, X: list[list[float]]) -> "StandardScaler":
        """Compute mean and std for each feature column."""
        n_features = len(X[0])
        self.means = []
        self.stds = []
        for col in range(n_features):
            column = [row[col] for row in X]
            mean = sum(column) / len(column)
            variance = sum((x - mean) ** 2 for x in column) / len(column)
            self.means.append(mean)
            self.stds.append(variance ** 0.5)
        self._fitted = True
        return self

    def transform(self, X: list[list[float]]) -> list[list[float]]:
        """Standardize data using fitted parameters."""
        if not self._fitted:
            raise RuntimeError("Scaler must be fitted before transform")
        result = []
        for row in X:
            scaled = [(x - m) / (s if s != 0 else 1.0)
                      for x, m, s in zip(row, self.means, self.stds)]
            result.append(scaled)
        return result

    def fit_transform(self, X: list[list[float]]) -> list[list[float]]:
        """Fit and transform in one call."""
        return self.fit(X).transform(X)

    def __repr__(self):
        return f"StandardScaler(fitted={self._fitted})"
Industry reality: this class won't work with scikit-learn's tooling. The hand-rolled StandardScaler above looks and behaves like scikit-learn's own scaler, but it can't be dropped into a Pipeline or tuned with GridSearchCV, because those tools rely on specific base-class behavior: get_params()/set_params() for hyperparameter introspection, and a consistent fit_transform() contract. Inheriting from sklearn.base.BaseEstimator and sklearn.base.TransformerMixin gives you both for free.

from sklearn.base import BaseEstimator, TransformerMixin
import numpy as np

class SklearnCompatibleScaler(BaseEstimator, TransformerMixin):
    """A custom transformer that plays nicely with Pipeline and GridSearchCV.

    BaseEstimator supplies get_params()/set_params() by introspecting
    the __init__ signature (so every __init__ argument must be stored
    as an identically-named attribute, unmodified).
    TransformerMixin supplies fit_transform() automatically, built
    from your fit() and transform().
    """

    def __init__(self, with_mean: bool = True):
        self.with_mean = with_mean  # stored as-is: BaseEstimator requires this

    def fit(self, X, y=None):  # y is accepted (and ignored) for Pipeline compatibility
        X = np.asarray(X)
        self.mean_ = X.mean(axis=0) if self.with_mean else np.zeros(X.shape[1])
        self.scale_ = X.std(axis=0)
        self.scale_[self.scale_ == 0] = 1.0
        return self

    def transform(self, X):
        X = np.asarray(X)
        return (X - self.mean_) / self.scale_

# Now this class is a first-class citizen in the sklearn ecosystem:
from sklearn.pipeline import Pipeline
from sklearn.model_selection import GridSearchCV
from sklearn.linear_model import LogisticRegression

pipe = Pipeline([
    ("scaler", SklearnCompatibleScaler()),
    ("clf", LogisticRegression()),
])
grid = GridSearchCV(pipe, param_grid={"scaler__with_mean": [True, False]}, cv=5)
grid.fit(X_train, y_train)  # GridSearchCV can now tune our custom transformer's hyperparameter

    @staticmethod
    def _euclidean(a, b):
        return math.sqrt(sum((x - y) ** 2 for x, y in zip(a, b)))

    def predict(self, X: list[list[float]]) -> list:
        """Predict labels for input samples."""
        predictions = []
        for sample in X:
            # Compute distances to all training points
            distances = [
                (self._euclidean(sample, train_x), train_y)
                for train_x, train_y in zip(self._X_train, self._y_train)
            ]
            # Sort by distance and take k nearest
            distances.sort(key=lambda x: x[0])
            k_nearest = [label for _, label in distances[:self.k]]
            # Majority vote
            from collections import Counter
            predictions.append(Counter(k_nearest).most_common(1)[0][0])
        return predictions

    def __repr__(self):
        return f"KNearestNeighbors(k={self.k})"

        }

5.6 A Gentler Step: Simple Inheritance Before PyTorch

super().__init__() can feel like a magic incantation the first time you see it inside nn.Module. Before jumping into PyTorch, look at what inheritance and super() are doing in a plain Python class with no framework involved:

class Model:
    """A minimal base class every model in our toolkit inherits from."""

    def __init__(self, name: str):
        self.name = name
        self.is_fitted = False

    def describe(self) -> str:
        return f"{self.name} (fitted={self.is_fitted})"


class LinearModel(Model):
    """Extends Model with the pieces specific to a linear model."""

    def __init__(self, name: str, n_features: int):
        super().__init__(name)          # runs Model.__init__, sets self.name and self.is_fitted
        self.n_features = n_features
        self.weights = [0.0] * n_features

    def fit(self, X, y):
        # ... fitting logic would go here ...
        self.is_fitted = True
        return self


lm = LinearModel(name="ols", n_features=3)
print(lm.describe())  # "ols (fitted=False)" -- describe() is inherited, unchanged
lm.fit(X=None, y=None)
print(lm.describe())  # "ols (fitted=True)"

Notice the pattern: LinearModel doesn't redefine describe() — it inherits it directly from Model. But it does need its own __init__, because it has extra state (n_features, weights) that the base class knows nothing about. Calling super().__init__(name) is what actually runs Model's setup code (setting self.name and self.is_fitted) before LinearModel adds its own attributes on top. If you skipped that call, self.name would never get set, and calling describe() would crash with an AttributeError.

This is exactly the same mechanism at work in nn.Module — the base class just does more setup behind the scenes (registering parameters, buffers, and submodules) instead of setting two simple attributes. Once this pattern feels natural in plain Python, the PyTorch version is just a bigger version of the same idea.

# Beam search for sequence generation
def beam_search_step(candidates, expand_fn, k):
    """One step of beam search using a heap.

    Args:
        candidates: Current beam of (score, sequence) tuples
        expand_fn: Function that generates next tokens
        k: Beam width
    """
    all_candidates = []
    for score, seq in candidates:
        for next_score, next_token in expand_fn(seq):
            # Length normalization prevents bias toward shorter sequences
            norm_score = (score * len(seq) + next_score) / (len(seq) + 1)
            all_candidates.append((norm_score, seq + [next_token]))
    return heapq.nlargest(k, all_candidates, key=lambda x: x[0])

6.5 Graphs for ML

from collections import deque, defaultdict

class Graph:
    """Adjacency-list graph for ML applications."""

    def __init__(self):
        self.adj = defaultdict(list)

    def add_edge(self, u, v):
        self.adj[u].append(v)
        self.adj[v].append(u)  # Undirected

    def bfs(self, start):
        """Breadth-first search. Used in GNN neighbor sampling."""
        visited = {start}
        queue = deque([start])
        order = []
        while queue:
            node = queue.popleft()
            order.append(node)
            for neighbor in self.adj[node]:
                if neighbor not in visited:
                    visited.add(neighbor)
                    queue.append(neighbor)
        return order

    def dfs(self, start):
        """Depth-first search. Used for connectivity analysis."""
        visited = set()
        stack = [start]
        order = []
        while stack:
            node = stack.pop()
            if node not in visited:
                visited.add(node)
                order.append(node)
                stack.extend(self.adj[node])
        return order

    def connected_components(self):
        """Find all connected components."""
        visited = set()
        components = []
        for node in self.adj:
            if node not in visited:
                comp = self._dfs_component(node, visited)
                components.append(comp)
        return components

    def _dfs_component(self, start, visited):
        comp = []
        stack = [start]
        while stack:
            node = stack.pop()
            if node not in visited:
                visited.add(node)
                comp.append(node)
                stack.extend(self.adj[node])
        return comp

6.6b Space Complexity and GPU VRAM

The "Memory" column above talks about big-O space complexity in the abstract, but on a GPU, memory is a hard, fixed budget — an A100 might give you 40GB or 80GB, period, and there's no swapping to disk gracefully when you run out. Interns frequently hit Out-Of-Memory (OOM) errors because they never connect the abstract complexity numbers to what's actually filling up VRAM. Four things typically consume it:

  • Model parameters: a model with $P$ parameters in FP32 needs $4P$ bytes just to store the weights (FP16/BF16 halves this).
  • Gradients: during training, you need another buffer the same size as the parameters to hold $\nabla L$ for every weight — roughly another $4P$ bytes.
  • Optimizer state: Adam keeps a running mean and variance per parameter, doubling the gradient cost again — Adam alone can need $8P$ bytes on top of the weights.
  • Activations: every intermediate tensor computed in the forward pass has to be kept around for the backward pass, and this scales with batch size and sequence length, not just model size — this is usually what actually causes OOM, not the weights themselves.
Diagnosing an OOM: if a model trains fine at batch size 8 but OOMs at batch size 32, the culprit is almost always activation memory, since it scales roughly linearly with batch size. The fix is usually one of: reduce batch size and use gradient accumulation to simulate the larger batch, enable mixed-precision training (FP16/BF16) to roughly halve activation memory, or use gradient checkpointing, which recomputes activations during the backward pass instead of storing them, trading extra compute for a large memory savings.

7.2 Dependency Management

Beyond requirements.txt. pip freeze > requirements.txt only records the packages you explicitly installed with the exact versions currently in your environment — it does not lock transitive dependencies (the dependencies of your dependencies) in a way that's reproducible across machines, and it provides no dependency resolution: two packages that need incompatible versions of a shared dependency will silently conflict or overwrite each other. This is a real headache in ML specifically, because the deep learning / data stack has an unusually deep and fast-moving dependency graph (CUDA versions, compiled C extensions, ABI compatibility between numpy/scipy/torch). Modern Python projects use a dependency manager that produces a full lock file instead.

# requirements.txt with pinned versions -- still common, but has the gaps noted above
numpy==1.24.3
pandas==2.0.3
scikit-learn==1.3.0
torch==2.0.1

pip freeze > requirements.txt
pip install -r requirements.txt

Poetry and uv are the two dominant modern alternatives. Both resolve your entire dependency graph up front and write it into a lock file that pins every package — direct and transitive — to an exact version and hash, so "works on my machine" becomes "works on every machine that installs from this lock file."

# Poetry: define dependencies in pyproject.toml, then:
poetry add numpy pandas scikit-learn torch
poetry lock              # resolves the full dependency graph -> poetry.lock
poetry install           # installs exactly what's in the lock file
poetry run python train.py

# uv: a much faster (Rust-based) alternative with a similar workflow
uv add numpy pandas scikit-learn torch
uv lock                  # writes uv.lock
uv sync                  # installs exactly what's in the lock file
uv run python train.py

The practical difference that matters day to day: uv resolves and installs dependencies dramatically faster than pip or Poetry (written in Rust, with aggressive caching), which matters when your CI pipeline is running pip install on every PR. Either tool is a solid choice over bare requirements.txt for anything beyond a quick throwaway script — for reproducible ML pipelines, prefer whichever your team has standardized on, and commit the lock file to version control alongside your code.

9.3 Data Contracts

A data contract defines expectations about your dataset. Models fail in production more often from input changes than algorithm issues.

from dataclasses import dataclass
from typing import List, Optional

@dataclass
class DataContract:
    """Define and validate dataset expectations."""
    required_columns: List[str]
    numeric_columns: List[str]
    categorical_columns: List[str]
    target_column: str
    categorical_values: dict  # column -> allowed values
    numeric_ranges: dict      # column -> (min, max)
    max_missing_ratio: float = 0.1

def validate(self, df: pd.DataFrame) -> dict:
    """Validate dataframe against contract. Returns issues dict."""
    issues = {}

    # Check required columns
    missing = set(self.required_columns) - set(df.columns)
    if missing:
        issues["missing_columns"] = list(missing)

    # Check numeric ranges
    for col, (min_val, max_val) in self.numeric_ranges.items():
        if col in df.columns:
            out_of_range = ((df[col] < min_val) | (df[col] > max_val)).sum()
            if out_of_range > 0:
                issues[f"{col}_out_of_range"] = out_of_range

    # Check categorical values
    for col, allowed in self.categorical_values.items():
        if col in df.columns:
            invalid = set(df[col].dropna().unique()) - set(allowed)
            if invalid:
                issues[f"{col}_invalid_values"] = list(invalid)

    # Check missing ratio
    missing_ratio = df[self.required_columns].isnull().mean()
    high_missing = missing_ratio[missing_ratio > self.max_missing_ratio]
    if len(high_missing) > 0:
        issues["high_missing"] = high_missing.to_dict()

    return issues

11.1b Timestamp-Bounded Joins: Preventing Temporal Leakage

The single most common way interns cause production incidents is a plain JOIN that silently pulls in a row from after the event they're trying to predict. A naive join like transactions JOIN accounts ON transactions.account_id = accounts.account_id attaches whatever the current account row looks like — including balance or status fields that were only updated after the transaction happened. In training, this leaks the future into the features; in production, that future data doesn't exist yet, so the model silently degrades.

The fix is an AS OF join (also called a timestamp-bounded join): explicitly constrain the join to only match rows whose timestamp precedes the event you're building features for. Some warehouses (e.g. Snowflake, KDB) have native ASOF JOIN syntax; in standard SQL you express the same constraint with an inequality join plus a window function to pick the most recent matching row:

-- Naive join (BUGGY): pulls whatever the account looks like *right now*,
-- which may include updates that happened after the transaction.
SELECT t.transaction_id, t.amount, a.account_status
FROM transactions t
JOIN accounts a ON t.account_id = a.account_id;   -- no time constraint!

-- Timestamp-bounded join: only match account snapshots that existed
-- at or before the transaction time.
SELECT t.transaction_id, t.amount, a.account_status
FROM transactions t
JOIN accounts a
    ON t.account_id = a.account_id
    AND a.account_update_timestamp <= t.transaction_timestamp   -- the critical guard
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY t.transaction_id
    ORDER BY a.account_update_timestamp DESC
) = 1;  -- keep only the most recent valid snapshot per transaction

-- Equivalent without QUALIFY (standard SQL, e.g. Postgres):
SELECT transaction_id, amount, account_status
FROM (
    SELECT
        t.transaction_id, t.amount, a.account_status,
        ROW_NUMBER() OVER (
            PARTITION BY t.transaction_id
            ORDER BY a.account_update_timestamp DESC
        ) AS rn
    FROM transactions t
    JOIN accounts a
        ON t.account_id = a.account_id
        AND a.account_update_timestamp <= t.transaction_timestamp
) ranked
WHERE rn = 1;
Rule of thumb: any time you join a fact table (events, transactions) against a slowly-changing dimension table (accounts, users, products) that gets updated over time, ask "which version of this row existed at the time of the event?" — not "what does this row look like today?" If the dimension table doesn't track history (no update timestamp per version), you cannot safely build point-in-time-correct features from it at all, and that's a data-engineering gap worth escalating before it becomes a silent leakage bug.

12.3 Linear Regression in Matrix Form

Geometric intuition first. Before looking at the formula, picture what OLS is actually doing. Every possible linear combination of the columns of $\mathbf{X}$ (i.e. every possible prediction $\mathbf{X}\mathbf{w}$) lies on a flat subspace called the column space of $\mathbf{X}$ — think of it as a flat plane sitting inside a higher-dimensional space. The target vector $\mathbf{y}$, however, generally does not lie on that plane, because real data is noisy. Least squares asks: which point on the plane is closest to $\mathbf{y}$? The answer is the orthogonal projection of $\mathbf{y}$ onto the column space — you drop a perpendicular from $\mathbf{y}$ straight down onto the plane. The leftover piece, $\mathbf{y} - \mathbf{X}\hat{\mathbf{w}}$ (the residual), is the error vector, and it is perpendicular to every column of $\mathbf{X}$. That perpendicularity condition, $\mathbf{X}^T(\mathbf{y} - \mathbf{X}\hat{\mathbf{w}}) = 0$, is exactly what you get if you expand it algebraically and solve for $\hat{\mathbf{w}}$.

Once that picture is in place, the closed-form solution is just algebra applied to the "residual is perpendicular to the columns" condition:

$$\hat{\mathbf{w}} = (\mathbf{X}^T \mathbf{X})^{-1} \mathbf{X}^T \mathbf{y}$$ 
import numpy as np

def linear_regression_closed_form(X, y):
    """Solve OLS using normal equations."""
    X_b = np.c_[np.ones((X.shape[0], 1)), X]  # Add bias column
    return np.linalg.inv(X_b.T @ X_b) @ X_b.T @ y

# Predict
w = linear_regression_closed_form(X_train, y_train)

12.4 Eigenvalues, Eigenvectors, and PCA

An eigenvector $\mathbf{v}$ of matrix $\mathbf{A}$ satisfies:

$$\mathbf{A}\mathbf{v} = \lambda \mathbf{v}$$

where $\lambda$ is the eigenvalue. Eigenvectors point in directions that the transformation only scales.

PCA finds the eigenvectors of the covariance matrix $\mathbf{X}^T\mathbf{X}$. These eigenvectors (principal components) are the directions of maximum variance.

def pca_from_scratch(X, n_components=2):
    """PCA via eigendecomposition of covariance matrix."""
    # Center data
    X_centered = X - X.mean(axis=0)

    # Covariance matrix
    cov = (X_centered.T @ X_centered) / (X.shape[0] - 1)

    # Eigendecomposition
    eigenvalues, eigenvectors = np.linalg.eigh(cov)

    # Sort by eigenvalue descending
    idx = np.argsort(eigenvalues)[::-1]
    eigenvalues = eigenvalues[idx]
    eigenvectors = eigenvectors[:, idx]

    # Project onto top components
    components = eigenvectors[:, :n_components]
    X_reduced = X_centered @ components

    return X_reduced, eigenvalues[:n_components], components

13.1 Derivatives and Gradients

Start in one dimension. Before the vector notation, look at the simplest possible case: $f(x) = x^2$, a parabola. At any point on the curve, the derivative $f'(x) = 2x$ tells you the slope of the line that just touches the curve there (the tangent). Where the curve is falling (left of the minimum), the slope is negative; where it's rising (right of the minimum), the slope is positive. If you always take a small step in the direction opposite the slope, you walk downhill toward the minimum no matter which side you start on — the derivative acts like a compass that always points toward lower values of $f$.

Now extend this to two dimensions. Instead of a curve, imagine a bowl-shaped surface (a mountain in reverse) and look down on it from above as a contour map, where each ring is a curve of constant height, just like on a topographic map. The gradient $\nabla f$ at a point is a vector that points in the direction of steepest ascent, always perpendicular to the contour line passing through that point. Gradient descent simply walks in the opposite direction — perpendicular to the contours, straight "downhill" — one small step at a time.

For multivariable functions, the gradient generalizes the single-variable derivative to a vector of partial derivatives, one per input dimension:

$$\nabla f = \left[\frac{\partial f}{\partial x_1}, \frac{\partial f}{\partial x_2}, \ldots, \frac{\partial f}{\partial x_d}\right]$$

13.2 Deriving Gradient Descent for Linear Regression

For linear regression with MSE loss:

$$L(\mathbf{w}) = \frac{1}{n} \sum_{i=1}^{n} (y_i - \mathbf{w}^T \mathbf{x}_i)^2$$

Taking the gradient:

$$\nabla L(\mathbf{w}) = -\frac{2}{n} \mathbf{X}^T (\mathbf{y} - \mathbf{X}\mathbf{w})$$

The gradient descent update:

$$\mathbf{w}_{t+1} = \mathbf{w}_t - \eta \nabla L(\mathbf{w}_t)$$ 
def gradient_descent(X, y, lr=0.01, epochs=1000):
    """Linear regression via gradient descent."""
    n, d = X.shape
    w = np.zeros(d)
    losses = []

    for epoch in range(epochs):
        # Forward pass
        y_pred = X @ w
        loss = np.mean((y - y_pred) ** 2)
        losses.append(loss)

        # Backward pass: compute gradient
        gradient = -(2/n) * X.T @ (y - y_pred)

        # Update
        w = w - lr * gradient

    return w, losses

13.3 Chain Rule and Backpropagation

Backpropagation is the chain rule applied to computational graphs. For a two-layer network:

$$\hat{y} = f_2(\mathbf{W}_2 \cdot f_1(\mathbf{W}_1 \mathbf{x}))$$

The gradient of loss $L$ with respect to $\mathbf{W}_1$ is:

$$\frac{\partial L}{\partial \mathbf{W}_1} = \frac{\partial L}{\partial \hat{y}} \cdot \frac{\partial \hat{y}}{\partial \mathbf{z}_2} \cdot \frac{\partial \mathbf{z}_2}{\partial \mathbf{h}} \cdot \frac{\partial \mathbf{h}}{\partial \mathbf{z}_1} \cdot \frac{\partial \mathbf{z}_1}{\partial \mathbf{W}_1}$$

14.2 Naive Bayes from a Probabilistic Perspective

Naive Bayes applies Bayes' theorem with the conditional independence assumption:

$$P(Y|X_1, \ldots, X_d) = \frac{P(Y) \prod_{i=1}^{d} P(X_i|Y)}{P(X_1, \ldots, X_d)}$$

The "naive" assumption is that features are conditionally independent given the class — rarely true, yet often effective.

14.4 Maximum Likelihood Estimation

MLE finds parameters that maximize the probability of observing the data:

$$\hat{\theta} = \arg\max_\theta P(X|\theta) = \arg\max_\theta \prod_{i=1}^{n} P(x_i|\theta)$$

For Gaussian MLE, this gives $\hat{\mu} = \frac{1}{n}\sum x_i$ and $\hat{\sigma}^2 = \frac{1}{n}\sum(x_i - \hat{\mu})^2$.

import numpy as np

def bootstrap_ci(data, statistic=np.mean, n_bootstrap=10000, ci=0.95):
    """Compute bootstrap confidence interval."""
    bootstrapped = []
    n = len(data)
    for _ in range(n_bootstrap):
        sample = np.random.choice(data, size=n, replace=True)
        bootstrapped.append(statistic(sample))

    lower = (1 - ci) / 2 * 100
    upper = (1 + ci) / 2 * 100
    return np.percentile(bootstrapped, [lower, upper])

from scipy import stats

def ab_test_z_score(conv_a, n_a, conv_b, n_b):
    """Two-proportion z-test for A/B comparison."""
    p_a = conv_a / n_a
    p_b = conv_b / n_b
    p_pooled = (conv_a + conv_b) / (n_a + n_b)

    se = (p_pooled * (1 - p_pooled) * (1/n_a + 1/n_b)) ** 0.5
    z = (p_a - p_b) / se
    p_value = 2 * (1 - stats.norm.cdf(abs(z)))

    return z, p_value

16.1 Entropy

Entropy measures the uncertainty in a random variable:

$$H(X) = -\sum_{i} P(x_i) \log P(x_i)$$

Higher entropy means more uncertainty. A fair coin has 1 bit of entropy; a certain outcome has 0.

16.2 Cross-Entropy Loss

The standard classification loss is cross-entropy between true distribution $y$ and predicted distribution $\hat{y}$:

$$L = -\sum_{i} y_i \log(\hat{y}_i)$$

For binary classification, this becomes:

$$L = -[y \log(\hat{y}) + (1-y) \log(1-\hat{y})]$$

16.3 KL Divergence

KL divergence measures how one distribution differs from another:

$$D_{KL}(P \| Q) = \sum_{i} P(x_i) \log \frac{P(x_i)}{Q(x_i)}$$

In VAEs, KL divergence regularizes the latent distribution to match a prior.

def cross_entropy(y_true, y_pred, eps=1e-15):
    """Numerically stable cross-entropy."""
    y_pred = np.clip(y_pred, eps, 1 - eps)
    return -np.mean(y_true * np.log(y_pred) +
                    (1 - y_true) * np.log(1 - y_pred))

def kl_divergence(p, q, eps=1e-10):
    """KL(P || Q)."""
    p = np.clip(p, eps, 1)
    q = np.clip(q, eps, 1)
    return np.sum(p * np.log(p / q))

20.3 Distribution Shift Types

Covariate shift: $P(X)$ changes but $P(Y|X)$ stays the same.
Label shift: $P(Y)$ changes but $P(X|Y)$ stays the same.
Concept drift: $P(Y|X)$ changes.

20.3b Data Drift vs. Concept Drift

In production monitoring, teams often collapse the taxonomy above into two practically distinct alarms, and it's worth being precise about the difference because the fix for each is completely different:

  • Data drift (covariate shift) is a change in the input distribution $P(X)$ — the kind of customers, transactions, or images the model now sees has shifted, but the underlying relationship between features and label hasn't changed. Example: an e-commerce fraud model trained mostly on desktop-browser traffic starts seeing a much higher share of mobile-app traffic after a marketing push. The features look different, but a fraudulent transaction is still fraudulent for the same underlying reasons.
  • Concept drift is a change in the relationship itself, $P(Y|X)$ — the same input now means something different. Example: a specific combination of purchase amount and location used to be a strong fraud signal, but fraudsters have since adapted their tactics, so that same combination is no longer predictive.

The practical distinction matters because data drift is detectable without labels — you can compare the distribution of incoming feature values against the training distribution (via a KS-statistic or PSI, as covered in Chapter 44) and flag a shift immediately. Concept drift is not detectable this way — the inputs can look completely unchanged while what they predict has quietly shifted, so you only catch it once ground-truth labels arrive and model performance visibly degrades (see the ground-truth delay discussion in Chapter 44). This is why production systems monitor both input distributions and (lagged) performance metrics — each catches a failure mode the other is blind to.

23.1 Linear Regression

Linear Regression

Problem: Predict a continuous target $y \in \mathbb{R}$ from features $\mathbf{x} \in \mathbb{R}^d$.

Core Intuition: Model the relationship as a hyperplane: $\hat{y} = \mathbf{w}^T \mathbf{x} + b$. Find weights that minimize the sum of squared errors.

Objective Function:

$$\min_{\mathbf{w}, b} \sum_{i=1}^{n} (y_i - (\mathbf{w}^T \mathbf{x}_i + b))^2$$

Assumptions: Linear relationship, independent errors with constant variance (homoscedasticity), no multicollinearity among features.

Computational Complexity: $O(nd^2 + d^3)$ for closed-form; $O(nd \cdot \text{epochs})$ for gradient descent.

Python Implementation:

from sklearn.linear_model import LinearRegression, Ridge, Lasso, ElasticNet
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

# With standardization (always scale for regularized models)
pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("reg", Ridge(alpha=1.0))
])
pipe.fit(X_train, y_train)
predictions = pipe.predict(X_test)

# Key hyperparameters:
# Ridge: alpha (regularization strength, default 1.0)
# Lasso: alpha (higher = more sparsity)
# ElasticNet: alpha, l1_ratio (0=Ridge, 1=Lasso)

Failure Modes:

  • Multicollinearity inflates coefficient variance (use Ridge or VIF analysis)
  • Outliers disproportionately affect fit (use Huber regression or RANSAC)
  • Extrapolation beyond training data is unreliable

When Not to Use: Non-linear relationships, categorical targets, highly correlated features without regularization.

23.2 Logistic Regression

Logistic Regression

Problem: Binary or multi-class classification.

Core Intuition: Model the log-odds as linear: $\log\frac{P(Y=1|X)}{P(Y=0|X)} = \mathbf{w}^T \mathbf{x} + b$. Apply sigmoid to get probabilities.

Logits vs. probabilities — a common point of confusion. The raw output of $\mathbf{w}^T\mathbf{x} + b$ is called the logit (or "raw score"). It is an unbounded real number — it can be $-50$ or $+50$ — and it is not a probability. Students frequently treat the logit as if it already were a probability, which breaks immediately: probabilities must lie in $[0, 1]$, and a raw score of $8.2$ means nothing on its own. The sigmoid function $\sigma(z) = \frac{1}{1 + e^{-z}}$ is what squashes that unbounded logit into a valid probability. A logit of $0$ maps to a probability of exactly $0.5$ (maximum uncertainty); large positive logits saturate toward $1$, large negative logits saturate toward $0$. This is why, in PyTorch, loss functions like BCEWithLogitsLoss expect the raw logit, not a pre-sigmoided probability — the "Logits" in the name is not decorative.

Objective Function (Cross-Entropy):

$$\min_{\mathbf{w}} -\sum_{i=1}^{n} [y_i \log(\hat{y}_i) + (1-y_i) \log(1-\hat{y}_i)] + \lambda \|\mathbf{w}\|_2^2$$

Python Implementation:

from sklearn.linear_model import LogisticRegression

model = LogisticRegression(
    C=1.0,              # Inverse regularization strength (smaller = stronger)
    penalty='l2',       # Regularization type: 'l1', 'l2', 'elasticnet'
    class_weight='balanced',  # Handle class imbalance
    max_iter=1000,
    solver='lbfgs'
)
model.fit(X_train, y_train)
probs = model.predict_proba(X_test)[:, 1]  # Probability of positive class

Calibration: Logistic regression outputs are well-calibrated probabilities by design. No post-hoc calibration needed in most cases.

23.3 Support Vector Machines

Support Vector Machine (SVM)

Problem: Classification or regression with margin maximization.

Core Intuition: Find the hyperplane that maximizes the distance (margin) to the nearest training points. Support vectors are the critical points that define the boundary.

Formally, the decision boundary is the set of points where $\mathbf{w}^T\mathbf{x} + b = 0$, and $\mathbf{w}$ is the vector orthogonal to that boundary — it points in the direction the classifier is most sensitive to. The two dashed margin lines are $\mathbf{w}^T\mathbf{x} + b = +1$ and $\mathbf{w}^T\mathbf{x} + b = -1$; the perpendicular distance between them works out to $\frac{2}{\|\mathbf{w}\|}$, which is exactly what SVM training maximizes (equivalently, it minimizes $\|\mathbf{w}\|$ subject to every point being correctly classified outside its margin).

Kernel Trick: Map data to higher dimensions implicitly using kernel functions:

Figure 23-3: The kernel trick. When data is not linearly separable in the original input space (left), the kernel trick implicitly maps it into a higher-dimensional feature space where a linear separator exists (right). The key insight is that SVM training and prediction only ever use dot products between data points $\phi(\mathbf{x})^T \phi(\mathbf{x}')$ — they never need the explicit mapping $\phi(\mathbf{x})$ itself. A kernel function $K(\mathbf{x}, \mathbf{x}')$ computes this dot product directly in the original space, giving the same result as if you had actually transformed the data, but without the computational cost of working in the (possibly infinite-dimensional) feature space. The RBF kernel, for instance, corresponds to an infinite-dimensional feature space, yet computing $K(\mathbf{x}, \mathbf{x}') = \exp(-\gamma \|\mathbf{x} - \mathbf{x}'\|^2)$ costs only $O(d)$ — the dimensionality of the original input.

  • Linear: $K(\mathbf{x}, \mathbf{x}') = \mathbf{x}^T \mathbf{x}'$
  • RBF: $K(\mathbf{x}, \mathbf{x}') = \exp(-\gamma \|\mathbf{x} - \mathbf{x}'\|^2)$
  • Polynomial: $K(\mathbf{x}, \mathbf{x}') = (\gamma \mathbf{x}^T \mathbf{x}' + r)^d$

Computational Complexity: Training $O(n^2 d)$ to $O(n^3 d)$; prediction $O(sv \cdot d)$ where $sv$ is number of support vectors.

from sklearn.svm import SVC, LinearSVC

# Linear SVM (fast for large datasets)
linear = LinearSVC(C=1.0, max_iter=10000)

# Kernel SVM (RBF for non-linear boundaries)
rbf = SVC(kernel='rbf', C=1.0, gamma='scale', probability=True)

# Key hyperparameters:
# C: Regularization (low = wider margin, more misclassifications)
# gamma: RBF kernel width (high = complex boundary, risk of overfitting)

When to Use: Small to medium datasets ($n < 100000$), high-dimensional data, when decision boundary shape matters.

When Not to Use: Very large datasets (training is slow), when probability estimates are critical (Platt scaling adds cost), datasets with many outliers.

24.1 Decision Trees

Decision Trees (CART)

Problem: Classification or regression via recursive partitioning.

Core Intuition: Split data on feature values to maximize purity (classification) or minimize variance (regression). Each leaf is a prediction.

Split Criteria:

  • Classification: Gini impurity $= 1 - \sum p_i^2$ or Entropy $= -\sum p_i \log p_i$
  • Regression: Mean Squared Error reduction

Walk through what "purest split" means concretely. Suppose a node has 10 samples, 5 of class A and 5 of class B — Gini impurity is $1 - (0.5)^2 - (0.5)^2 = 0.5$, the maximum possible for two classes (pure guesswork). Now suppose a candidate split on some feature sends 6 samples (5 A, 1 B) left and 4 samples (0 A, 4 B) right. The left child's impurity is $1 - (5/6)^2 - (1/6)^2 \approx 0.278$; the right child is perfectly pure at $1 - 1^2 - 0^2 = 0$. The split's overall impurity is the sample-weighted average of the children: $\frac{6}{10}(0.278) + \frac{4}{10}(0) \approx 0.167$ — a large drop from the parent's $0.5$. The tree evaluates every feature and every possible threshold this way and greedily picks whichever split produces the lowest weighted-average impurity, i.e. the largest Gini gain over the parent.

from sklearn.tree import DecisionTreeClassifier, export_text

model = DecisionTreeClassifier(
    criterion='gini',       # or 'entropy', 'log_loss'
    max_depth=5,             # Prevent overfitting (critical!)
    min_samples_split=20,    # Minimum samples to split a node
    min_samples_leaf=10,     # Minimum samples in a leaf
    max_features='sqrt',     # Consider sqrt(n_features) per split
    random_state=42
)
model.fit(X_train, y_train)

# Interpretability: print tree rules
print(export_text(model, feature_names=feature_names))

When Not to Use: Unconstrained trees overfit severely. Always set max_depth, min_samples_leaf, or use ensemble methods.

24.2 Random Forest

Figure 24-2: Random Forest (bagging). Each tree is trained independently on a different bootstrap sample (sampling with replacement from the training data) and considers a random subset of features at each split. Because the trees are trained in parallel and each sees a slightly different view of the data, their errors are largely uncorrelated — averaging across many such trees cancels out the individual trees' overfitting, dramatically reducing variance while keeping bias low. The two sources of randomness (bootstrap sampling + feature subsampling) are what make Random Forests work: without them, all trees would be identical and averaging would provide no benefit.

Random Forest

Core Intuition: Train many decision trees on bootstrapped samples with random feature subsets, then aggregate predictions. Bagging reduces variance.

from sklearn.ensemble import RandomForestClassifier

model = RandomForestClassifier(
    n_estimators=100,     # Number of trees
    max_depth=10,
    max_features='sqrt',  # Random subset of features per split
    n_jobs=-1,            # Use all CPU cores
    random_state=42
)
model.fit(X_train, y_train)

# Feature importance (use with caution: biased toward high-cardinality features)
importances = pd.DataFrame({
    'feature': feature_names,

Strengths: Excellent baseline for tabular data, handles mixed types, robust to outliers, provides feature importance, parallelizable.

24.3 Gradient Boosting

Figure 24-3: Gradient Boosting (sequential). Unlike Random Forest's parallel trees, gradient boosting builds trees one at a time, where each new tree is trained to predict the residuals (errors) of the current ensemble. The ensemble prediction after $m$ trees is $F_m(x) = F_{m-1}(x) + \eta \cdot h_m(x)$, where $\eta$ is the learning rate (shrinkage) and $h_m$ is the $m$-th tree. This is gradient descent in function space: each tree takes a step in the direction that reduces the loss, with the learning rate controlling the step size. The sequential nature means boosting cannot be trivially parallelized like Random Forest, but it typically achieves lower bias at the cost of being more prone to overfitting if the learning rate is too high or too many trees are used.

Gradient Boosting (XGBoost, LightGBM, CatBoost)

Core Intuition: Train trees sequentially, where each new tree predicts the residuals (errors) of the previous ensemble. Gradient descent in function space.

Early stopping, concretely. Gradient boosting adds one tree at a time, and each new tree reduces training error further — left unchecked, it will eventually memorize the training set. Early stopping holds out a validation set, tracks the validation metric after every boosting round, and halts training once that metric stops improving for a fixed number of rounds (the "patience" window), keeping the model from the best-performing round rather than the last one.

25.1 K-Means Clustering

Figure 25-1: K-Means iteration (Expectation-Maximization). The algorithm alternates between two steps: (1) the assignment step assigns each data point to the nearest centroid (the E-step), and (2) the update step recomputes each centroid as the mean of its assigned points (the M-step). This is guaranteed to decrease the within-cluster sum of squares at every iteration, so the algorithm always converges — but only to a local minimum. The final result depends heavily on the initial centroid placement, which is why K-Means++ (a smarter initialization that spreads centroids apart) and multiple random restarts (n_init=10) are standard practice.

K-Means

Problem: Partition $n$ observations into $k$ clusters by minimizing within-cluster sum of squares.

Algorithm:

  1. Initialize $k$ centroids randomly
  2. Assign each point to nearest centroid
  3. Update centroids as mean of assigned points
  4. Repeat until convergence

Objective:

$$\min_{\mu, C} \sum_{i=1}^{k} \sum_{x \in C_i} \|x - \mu_i\|^2$$ 
from sklearn.cluster import KMeans

model = KMeans(n_clusters=3, random_state=42, n_init=10)
labels = model.fit_predict(X)
centers = model.cluster_centers_

# Elbow method for choosing k
inertias = []
for k in range(1, 10):
    km = KMeans(n_clusters=k, random_state=42, n_init=10).fit(X)
    inertias.append(km.inertia_)

# Plot elbow curve to select optimal k

Assumptions: Spherical clusters, similar sizes, similar densities. Uses Euclidean distance.

When Not to Use: Non-convex cluster shapes, clusters of very different sizes/densities, categorical data, high-dimensional sparse data.

25.2 DBSCAN

DBSCAN

Core Intuition: Groups points that are closely packed, marking points in low-density regions as noise. Discovers clusters of arbitrary shape.

from sklearn.cluster import DBSCAN

model = DBSCAN(eps=0.5, min_samples=5)
labels = model.fit_predict(X)

# labels == -1 indicates noise points
n_noise = list(labels).count(-1)
n_clusters = len(set(labels)) - (1 if -1 in labels else 0)

Parameters:

  • eps ($\epsilon$): Neighborhood radius. Too small = many noise points; too large = one big cluster
  • min_samples: Minimum points to form a dense region. Higher = more noise, fewer clusters

Strengths: Arbitrary cluster shapes, identifies outliers, no need to specify $k$.

Weaknesses: Sensitive to eps parameter, struggles with varying densities, slow on very large datasets ($O(n^2)$ without spatial index).

25.3 Hierarchical Clustering

Hierarchical (Agglomerative)

Core Intuition: Build a tree of clusters (dendrogram) by iteratively merging the closest pairs.

  • Complete: Maximum distance. Creates compact clusters.
  • Average: Mean distance between all pairs. Balanced.

28.1 Markov Decision Processes

An MDP is defined by $(S, A, P, R, \gamma)$:

  • $S$: Set of states
  • $A$: Set of actions
  • $P(s'|s, a)$: Transition probability
  • $R(s, a, s')$: Reward function
  • $\gamma \in [0, 1]$: Discount factor

Bellman Optimality Equation:

$$V^*(s) = \max_a \sum_{s'} P(s'|s,a)[R(s,a,s') + \gamma V^*(s')]$$

28.2 Q-Learning

Model-free RL that learns action-values $Q(s, a)$:

$$Q(s_t, a_t) \leftarrow Q(s_t, a_t) + \alpha[r_t + \gamma \max_a Q(s_{t+1}, a) - Q(s_t, a_t)]$$ 
import numpy as np

class QLearningAgent:
    def __init__(self, n_states, n_actions, lr=0.1, gamma=0.99, epsilon=0.1):
        self.q_table = np.zeros((n_states, n_actions))
        self.lr = lr
        self.gamma = gamma
        self.epsilon = epsilon
        self.n_actions = n_actions

    def select_action(self, state):
        if np.random.random() < self.epsilon:
            return np.random.randint(self.n_actions)
        return np.argmax(self.q_table[state])

    def update(self, state, action, reward, next_state):
        best_next = np.max(self.q_table[next_state])
        td_target = reward + self.gamma * best_next
        td_error = td_target - self.q_table[state, action]
        self.q_table[state, action] += self.lr * td_error

29.1 Neural Network Basics

Figure 29-0: Multi-layer perceptron (MLP). Each node in a hidden layer computes $h = \text{activation}(W \cdot \text{input} + b)$, where $W$ and $b$ are learned parameters. Every node in one layer connects to every node in the next, making the transformation a sequence of matrix multiplications interleaved with nonlinear activations. Without the nonlinear activations, the entire network would collapse to a single linear transformation (composition of linear maps is linear). The hidden layers successively transform the input into increasingly abstract representations, with earlier layers capturing simple patterns (edges, intervals) and deeper layers composing them into complex features.

Start with one neuron as a computation graph. Before stacking layers, see exactly what a single artificial neuron computes. It takes inputs $x_1, x_2, \ldots$, multiplies each by a learned weight, sums the results together with a learned bias $b$, and then passes that sum through a non-linear activation function:

Concretely, for two inputs: $z = x_1 w_1 + x_2 w_2 + b$, then the neuron's output is $a = \text{ReLU}(z) = \max(0, z)$. Every "layer" in a neural network is just many of these neurons computed in parallel (as a matrix multiply $\mathbf{W}\mathbf{x} + \mathbf{b}$), and a "deep" network is this block repeated many times, each layer's output feeding the next layer's input. This is the graph that backpropagation walks in reverse to compute gradients — every $+$, $\times$, and activation is a node with a known local derivative, and the chain rule strings them together.

A neural network is a composition of linear transformations and non-linear activations:

$$\hat{y} = f_L(W_L \cdot f_{L-1}(W_{L-1} \cdots f_1(W_1 \mathbf{x} + b_1) \cdots + b_{L-1}) + b_L)$$

29.3 PyTorch Training Loop

Figure 29-2: The PyTorch training loop. Each iteration follows five steps: (1) load a mini-batch from the DataLoader, (2) compute the forward pass through the model to get predictions, (3) compute the loss function comparing predictions to targets, (4) call loss.backward() to populate every parameter's .grad attribute via backpropagation, and (5) call optimizer.step() to update the weights using those gradients. Crucially, optimizer.zero_grad() must be called before the next backward pass, because PyTorch accumulates gradients rather than overwriting them.

Why optimizer.zero_grad() is not optional. PyTorch accumulates gradients into the .grad attribute of every parameter every time you call .backward() — it adds to whatever was already there rather than overwriting it. This is a deliberate design choice that makes gradient accumulation over multiple mini-batches easy, but it means that if you forget to zero the gradients before the next batch's backward pass, you'll silently sum gradients across batches and the optimizer will take wildly wrong steps. optimizer.zero_grad() must be called once per batch, before loss.backward(). 
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset

# Model definition
class MLP(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, output_dim)
        )

    def forward(self, x):
        return self.net(x)

# --- Training step ---
def train_epoch(model, dataloader, optimizer, criterion, device):
    model.train()  # enables dropout / batchnorm training behavior
    total_loss = 0
    for batch_x, batch_y in dataloader:
        batch_x, batch_y = batch_x.to(device), batch_y.to(device)

        optimizer.zero_grad()          # clear accumulated gradients from the last batch
        outputs = model(batch_x)
        loss = criterion(outputs, batch_y)
        loss.backward()                # accumulate fresh gradients into .grad

        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        optimizer.step()               # apply the update using those gradients
        total_loss += loss.item()
    return total_loss / len(dataloader)

# --- Validation step ---
def validate_epoch(model, dataloader, criterion, device):
    model.eval()  # disables dropout, freezes batchnorm running stats
    total_loss, correct, total = 0, 0, 0
    with torch.no_grad():  # no autograd graph needed -> saves memory and compute
        for batch_x, batch_y in dataloader:
            batch_x, batch_y = batch_x.to(device), batch_y.to(device)
            outputs = model(batch_x)
            loss = criterion(outputs, batch_y)
            total_loss += loss.item()
            correct += (outputs.argmax(dim=1) == batch_y).sum().item()
            total += batch_y.size(0)
    return total_loss / len(dataloader), correct / total

# Setup
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
model = MLP(input_dim=784, hidden_dim=256, output_dim=10).to(device)
criterion = nn.CrossEntropyLoss()
optimizer = optim.AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=100)

# Convert numpy to torch tensors and create dataloaders
train_ds = TensorDataset(torch.FloatTensor(X_train), torch.LongTensor(y_train))
val_ds = TensorDataset(torch.FloatTensor(X_val), torch.LongTensor(y_val))
train_loader = DataLoader(train_ds, batch_size=64, shuffle=True)
val_loader = DataLoader(val_ds, batch_size=64, shuffle=False)

# Training with validation each epoch
best_val_loss = float('inf')
for epoch in range(100):
    train_loss = train_epoch(model, train_loader, optimizer, criterion, device)
    val_loss, val_acc = validate_epoch(model, val_loader, criterion, device)
    scheduler.step()

    if val_loss < best_val_loss:
        best_val_loss = val_loss
        torch.save(model.state_dict(), "best_model.pt")  # keep the best checkpoint, not the last

30.2 Modern CNN Architectures

Figure 30-1: ResNet residual block. The key innovation is the skip connection that adds the block's input directly to its output. Without the shortcut, deep networks often degrade in training accuracy — not from overfitting, but from the difficulty of learning identity mappings through many nonlinear layers. The skip connection gives the network an easy path to learn the identity (just zero out the residual), and when the block does learn a useful transformation, the network can learn the residual $F(x) = H(x) - x$ instead of $H(x)$ directly, which is empirically easier to optimize. This simple trick enabled training networks with 100+ layers and is the foundation of nearly all modern CNN architectures.

31.1 RNNs, LSTMs, and GRUs

RNN: Processes sequences by maintaining a hidden state updated at each timestep. Suffers from vanishing gradients in long sequences.

Figure 31-1: Unrolled RNN. At each timestep $t$, the RNN cell receives the current input $x_t$ and the previous hidden state $h_{t-1}$, producing output $y_t$ and the updated hidden state $h_t$ that is passed to the next timestep. Backpropagation through time (BPTT) unrolls this graph and applies the chain rule across all timesteps, but repeated multiplication by the same weight matrix causes gradients to either vanish (shrink exponentially) or explode (grow exponentially) over long sequences.

LSTM: Uses gating mechanisms (input, forget, output gates) to selectively remember/forget information.

Figure 31-2: LSTM cell. The cell state (horizontal highway at top) carries information across timesteps with minimal interference — the forget gate multiplies it elementwise by $f_t \in (0,1)$ to erase information, and the input gate adds new candidate values scaled by $i_t$. The output gate controls how much of the cell state is exposed as the hidden state $h_t$. This highway-structure gradient path allows the network to learn long-range dependencies without suffering from vanishing gradients, because the cell state gradient can flow through the + node unchanged (addition has gradient 1).

GRU: Simplified gating with reset and update gates. Fewer parameters than LSTM, often comparable performance.

32.1 The Transformer Architecture

Key components:

  • Self-attention: Each token attends to all others. Computes Query-Key-Value attention.
  • Multi-head attention: Multiple attention operations in parallel.
  • Position encoding: Injects sequence position information.
  • Feed-forward network: Applied independently to each token.
  • Layer normalization + residual connections: Stabilize deep training.

Figure 32-0: The full Transformer architecture (after Vaswani et al., "Attention Is All You Need", 2017). The encoder (left) processes the input sequence through N identical blocks, each containing multi-head self-attention followed by a feed-forward network, with residual connections and layer normalization around each sub-layer. The decoder (right) adds a cross-attention sub-layer that attends to the encoder's output: the encoder's final representations become the Keys and Values, while the decoder's self-attention output provides the Queries. The decoder's first self-attention is masked to prevent attending to future positions, enforcing left-to-right (causal) generation. The final linear projection and softmax produce a probability distribution over the output vocabulary at each position. Modern decoder-only models (GPT, LLaMA) drop the encoder entirely and use only the masked self-attention stack; encoder-only models (BERT) drop the decoder and use bidirectional (unmasked) self-attention.

32.2 Self-Attention Mechanism

The database / search-engine analogy. Before the formula, think of self-attention as a soft lookup in a search engine. Every token produces three vectors from its embedding: a Query ("what am I looking for?"), a Key ("what do I have to offer, as a searchable label?"), and a Value ("what content do I actually contain?"). To decide what a token should pay attention to, you compare its Query against every other token's Key — like typing a search query and matching it against article titles. Tokens whose Keys match well get a high score; that score, after softmax, becomes a weight. The output is a weighted blend of every token's Value, weighted by how well its Key matched the Query — like retrieving a mix of article contents, more from the best-matching articles and less from the rest.

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

Scaled dot-product attention prevents softmax saturation for large $d_k$.

Worked example: a 3-word sentence. Take the sentence "cats chase mice" with $d_k = 2$ for simplicity. Each word's embedding is projected through learned weight matrices $W_Q, W_K, W_V$ to produce three $3 \times 2$ matrices $Q$, $K$, $V$ (one row per word). Computing $QK^T$ gives a $3 \times 3$ matrix of raw similarity scores — row $i$, column $j$ is how strongly word $i$'s query matches word $j$'s key (e.g. "chase" will likely score highly against both "cats" and "mice", since a verb needs to know its subject and object). Dividing by $\sqrt{d_k}$ keeps these scores in a numerically stable range, and applying softmax row-wise turns each row into a probability distribution that sums to 1. Multiplying by $V$ then replaces each word's representation with a weighted average of all three words' Values, using those per-row probabilities as the weights — "chase" ends up with a representation that blends in information from "cats" and "mice" in proportion to how relevant they are.

Figure 32-3: Word embeddings as the foundation of retrieval. Before Transformers can process text, words must be converted to dense vector representations. The Word2Vec diagram above shows the core insight: semantic relationships are encoded as geometric directions in vector space, so King − Man + Woman ≈ Queen. Modern embedding models (like the ones used in RAG pipelines) extend this principle: they map entire passages to vectors in a shared space, so that a query vector's nearest neighbors are the most semantically relevant passages. The RAG pipeline then retrieves these passages and injects them into the LLM's prompt as context — grounding the generated answer in retrieved evidence rather than parametric memory alone.

  1. Index documents in a vector database
  2. Retrieve relevant chunks for a query
  3. Include retrieved context in the prompt
  4. Generate answer grounded in retrieved text

Interns are mostly shipping GenAI wrappers, and RAG is the piece that determines whether those wrappers hallucinate or actually work. It has three moving parts worth understanding in depth.

Embedding models turn text into fixed-length vectors such that semantically similar text ends up close together in vector space. In production you will pick between hosted embedding APIs (e.g. OpenAI's text-embedding-3, Cohere Embed) and open-weight models you can self-host (e.g. BGE, E5, or Sentence-Transformers models). The key trade-offs are dimensionality (higher dimensions cost more storage and compute per query but can capture more nuance), context window (can it embed a full paragraph or only a sentence), and whether the model was tuned for retrieval (query-document pairs) versus general similarity.

Vector databases store those embeddings and answer "find me the $k$ nearest vectors to this query vector" efficiently, typically using an approximate nearest-neighbor index (HNSW is the most common) rather than brute-force search, which would not scale past a few thousand documents. Popular choices include managed services like Pinecone and self-hostable options like Milvus, Weaviate, Qdrant, or even pgvector as a Postgres extension when you don't want to run a separate system.

# Minimal RAG retrieval step (conceptual, using a Pinecone-style client)

    vector=query_embedding,
    top_k=5,
    include_metadata=True,
)

Chunking strategy is where most RAG quality problems actually live. If chunks are too large, irrelevant text dilutes the embedding and the retriever misses the query's intent; too small, and a chunk loses the surrounding context needed to answer correctly. Recursive character chunking is the practical default: it tries to split on the largest available semantic boundary first (paragraph breaks), and only falls back to smaller boundaries (sentences, then words, then raw characters) if a piece is still too large, so chunks tend to respect natural document structure rather than cutting mid-sentence. A typical starting point is 300–800 tokens per chunk with a 10–20% overlap between consecutive chunks, so a fact that happens to sit at a chunk boundary isn't lost.

Common failure mode: chunking by a fixed character count with no overlap and no respect for paragraph boundaries. This silently splits sentences and tables in half, and the retriever will confidently return half-facts. Always inspect a sample of your actual chunks before shipping a RAG pipeline.

33.2 Message Passing

Figure 33-1: GNN message passing. At each layer, a node $i$ collects messages from its neighbors $\mathcal{N}(i)$, aggregates them (typically by sum, mean, or max), and combines the aggregated message with its own current representation via an update function (often a GRU or MLP). After $L$ layers, each node's representation incorporates information from its $L$-hop neighborhood — a node "sees" all nodes within $L$ edges. Different GNN variants (GCN, GraphSAGE, GAT) differ in how they define the message function and the aggregation: GCN uses a normalized weighted sum, GraphSAGE samples a fixed-size neighborhood and concatenates, and GAT uses learned attention weights to differentially attend to each neighbor.

GNNs operate through message passing:

$$\mathbf{h}_i^{(l+1)} = \text{UPDATE}\left(\mathbf{h}_i^{(l)}, \text{AGGREGATE}_{j \in \mathcal{N}(i)}(\mathbf{h}_j^{(l)})\right)$$

Each node updates its representation by aggregating messages from neighbors.

34.1 Autoencoder Types

Figure 34-1: Autoencoder vs. VAE. A standard autoencoder (top) maps the input through a bottleneck that forces the network to learn a compressed representation — the latent vector $z$ is a deterministic function of $x$. A variational autoencoder (bottom) replaces the deterministic bottleneck with a probabilistic one: the encoder outputs parameters $\mu$ and $\log\sigma^2$ of a Gaussian distribution $q(z|x)$, and $z$ is sampled from this distribution via the reparameterization trick ($z = \mu + \sigma \cdot \epsilon$, where $\epsilon \sim \mathcal{N}(0, I)$). This trick is what makes the VAE trainable: the random sampling is moved outside the computation graph (via $\epsilon$), so gradients can flow through $\mu$ and $\sigma$ even though $z$ is stochastic. The KL divergence term in the VAE loss pushes $q(z|x)$ toward the prior $\mathcal{N}(0, I)$, ensuring the latent space is smooth and continuous — which is what enables sampling and interpolation.

34.2 Variational Autoencoder (VAE)

VAE learns a probabilistic latent space:

$$\mathcal{L} = \underbrace{\mathbb{E}_{q(z|x)}[\log p(x|z)]}_{\text{Reconstruction}} - \underbrace{D_{KL}(q(z|x) \| p(z))}_{\text{KL Regularization}}$$

The KL term forces the learned distribution to match the prior (typically $\mathcal{N}(0, I)$), enabling sampling and interpolation.

34.3 Anomaly Detection with Autoencoders

Approach: Train autoencoder on normal data. Anomalies have high reconstruction error.

Limitation: Autoencoder anomaly detection assumes anomalies are harder to reconstruct. This may not hold: a simple outlier might be reconstructed perfectly while a complex normal pattern has higher error. Always validate with labeled anomalies.

36.2 Diffusion Models

Figure 36-0: Diffusion forward and reverse processes. The forward process (left-to-right) is a fixed, no-learning-required recipe: at each step, a small amount of Gaussian noise is added according to a variance schedule $\beta_1, \ldots, \beta_T$, gradually destroying the image structure until only noise remains. The reverse process (right-to-left) is what the neural network learns: given a noisy image $x_t$ and the timestep $t$, predict the noise $\epsilon$ that was added, then subtract it to step back toward $x_{t-1}$. Generation means starting from pure noise $x_T \sim \mathcal{N}(0, I)$ and running the reverse process for $T$ steps. The training objective is simple: sample a clean image $x_0$, sample a timestep $t$, add noise to get $x_t$, and train the network to predict the noise $\epsilon$ that was added — a regression problem.

Picture it before the equations. Diffusion models are built around a process that is easy to visualize: take a clean image, and over many small steps, add a little bit of random (Gaussian) noise each time. After enough steps, the "image" is indistinguishable from pure static — all of the original structure has been destroyed. That's the forward process, and it needs no learning at all; it's just a fixed recipe for adding noise. The interesting part is the reverse process: a neural network is trained to look at a noisy image at step $t$ and predict either the noise that was added, or a slightly cleaner version at step $t-1$. Run that denoising network repeatedly, starting from pure random noise, and it gradually "sculpts" a coherent image out of static — generation is just denoising run backwards from noise to signal.

The neural network's actual job during training is deceptively simple: given a noisy image $x_t$ and the timestep $t$, predict the noise vector $\epsilon$ that was mixed in to produce it. Once it can do that reliably at every noise level, subtracting the predicted noise (scaled appropriately) is exactly what walks the sample one step back toward $x_{t-1}$ — repeat that $T$ times starting from pure noise $x_T$, and you arrive at a sample from the data distribution, $x_0$.

Forward process: Gradually add Gaussian noise over $T$ steps:

$$q(x_t | x_{t-1}) = \mathcal{N}(x_t; \sqrt{1 - \beta_t} x_{t-1}, \beta_t I)$$

Reverse process: Learn to denoise:

$$p_\theta(x_{t-1} | x_t) = \mathcal{N}(x_{t-1}; \mu_\theta(x_t, t), \Sigma_\theta(x_t, t))$$

Sampling parameters:

  • Steps: More steps = higher quality, slower generation. Typical: 20-50 (DDIM) to 1000 (DDPM)
  • Guidance scale: Higher = stronger prompt adherence, less diversity. Typical: 3-10 (Classifier-Free Guidance)

36.3 Autoregressive Models

Generate by predicting one token at a time. Key parameters:

  • Temperature: Scales logits before softmax. Higher = more random; Lower = more deterministic.
  • Top-k: Sample only from top k most likely tokens.
  • Top-p (nucleus): Sample from smallest set of tokens whose cumulative probability exceeds p.
def sample_top_p(logits, p=0.9, temperature=1.0):
    probs = F.softmax(logits / temperature, dim=-1)
    sorted_probs, sorted_indices = torch.sort(probs, descending=True)
    cumsum = torch.cumsum(sorted_probs, dim=-1)

    # Remove tokens with cumulative probability above threshold
    mask = cumsum > p
    mask[..., 0] = False  # Keep at least one token
    sorted_probs[mask] = 0
    sorted_probs /= sorted_probs.sum()

    idx = torch.multinomial(sorted_probs, 1)
    return sorted_indices[idx]

Figure 37-1: Three causal structures. A confounder (shown in Figure 37-1a) is a common cause of both treatment and outcome — it creates a spurious (non-causal) association between them, which is why observational studies that fail to adjust for confounders overestimate causal effects. 38.1 Why Explainability Matters

Machine learning models are increasingly deployed in high-stakes domains — medical diagnosis, credit lending, criminal sentencing, autonomous driving — where a wrong prediction can harm people. In these settings, accuracy alone is insufficient; stakeholders need to understand why a model made a particular decision. Explainable AI (XAI) is the research area and engineering practice of making model behavior understandable to humans.

There are three distinct motivations for explainability, and understanding which one applies to your use case determines which methods you should reach for:

  • Regulatory compliance: The EU's General Data Protection Regulation (GDPR) includes a "right to explanation" for automated decisions. The US Equal Credit Opportunity Act (ECOA) requires lenders to provide specific reasons for adverse credit decisions. If you work in finance or healthcare, explainability is not optional — it is a legal requirement, and only certain methods (like SHAP or LIME) produce the kind of per-patient, per-applicant attribution that regulators accept.
  • Model debugging and improvement: When a model underperforms on a subgroup, explanations can reveal why — perhaps it is relying on a spurious feature (like using the hospital ID as a proxy for disease severity), or perhaps a feature is missing important interactions. Feature importance and partial dependence plots are debugging tools first and trust-building tools second.
  • User trust and adoption: A radiologist who sees a heatmap highlighting the region of a scan that drove the model's cancer prediction is more likely to trust (or appropriately challenge) the model than one who receives a bare probability score. Saliency maps and Grad-CAM serve this purpose for vision models; SHAP waterfall plots serve it for tabular models.
Intrinsic vs. post-hoc explainability. Some models are inherently interpretable — you can read off the decision logic directly. Linear regression coefficients, decision tree paths, and naive Bayes conditional probabilities are all intrinsic explanations: the model is the explanation. Most modern models (neural networks, gradient boosting, ensemble methods) are black boxes that require post-hoc explanation methods applied after training. The gold standard, when interpretability is a primary requirement, is to use an intrinsically interpretable model. When accuracy demands a black box, post-hoc XAI methods are the next best thing — but they are approximations of the truth, not the truth itself.

38.2 XAI Taxonomy

Explanation methods can be classified along three axes: scope (global vs. local), model dependence (agnostic vs. specific), and timing (intrinsic vs. post-hoc). Understanding these axes is essential for choosing the right tool:

  • Global explanations describe the model's overall behavior — which features matter most across all predictions, how each feature affects the output on average. These are useful for model debugging, feature selection, and regulatory reporting. Examples: permutation importance, partial dependence plots (PDP), accumulated local effects (ALE).
  • Local explanations describe why the model made a specific prediction for a specific input. These are useful for individual decision justification — explaining to a patient why they were flagged as high-risk, or to an applicant why their loan was denied. Examples: SHAP, LIME, counterfactuals, saliency maps.
  • Model-agnostic methods work with any model because they only require the ability to query the model with inputs and observe outputs. They treat the model as a black box. Examples: SHAP, LIME, permutation importance, PDP.
  • Model-specific methods exploit the internal structure of particular model types. Saliency maps require gradient access (neural networks only); tree-based feature importance exploits the split structure of decision trees. These methods are typically faster and more faithful to the model's actual computation, but they cannot be transferred to other model types.

38.3.1 Permutation Feature Importance

Permutation importance measures how much a model's performance degrades when a single feature's values are randomly shuffled across the validation set, breaking the relationship between that feature and the target. If shuffling a feature causes a large drop in accuracy (or increase in loss), that feature was important; if shuffling has no effect, the model was not relying on it. The algorithm is straightforward: (1) evaluate the model on the validation set to get a baseline score, (2) for each feature, shuffle its column, re-evaluate, and record the score drop, (3) rank features by score drop. Because it only requires model predictions (not internals), it works with any model.

Key caveat: Permutation importance can be misleading when features are highly correlated. If features A and B are both important and correlated, shuffling A alone may not hurt performance much because B still carries the same information — the model simply "reads" the signal from B instead. This makes both A and B appear unimportant when they are actually both critical. Accumulated Local Effects (ALE) plots address this issue.

from sklearn.inspection import permutation_importance

    model, X_val, y_val,
    n_repeats=10,         # Number of shuffles per feature
    random_state=42,
    scoring='accuracy'    # Or 'neg_log_loss', 'f1', etc.
)

# Display results

38.3.2 Partial Dependence Plots (PDP) and ALE

A Partial Dependence Plot shows the average predicted response as a function of one (or two) feature values, marginalizing over all other features. For a feature $x_S$, the partial dependence function is:

$$PD(x_S) = \frac{1}{n} \sum_{i=1}^{n} f(x_S, \mathbf{x}_{C}^{(i)})$$

where $\mathbf{x}_{C}^{(i)}$ are the values of all other features for the $i$-th sample. We fix $x_S$ at each value in a grid, replace the other features with actual data points, average the predictions, and plot the curve. A rising PDP means the feature increases the prediction; a flat PDP means the feature has no average effect.

PDP's critical flaw: the marginalization assumption. PDP averages over the marginal distribution of the complement features, which means it evaluates the model at feature combinations that may never occur in reality. If income and education are strongly correlated, PDP will evaluate the model at low-income + PhD — a combination that barely exists in the data. This creates unrealistic "data points" that can distort the plot. ALE (Accumulated Local Effects) fixes this by computing effects conditional on the actual joint distribution: it averages only over nearby observations, avoiding extrapolation. ALE should be preferred whenever features are correlated (which is almost always the case with real data).

from sklearn.inspection import PartialDependenceDisplay

# 1D PDP for top features
fig, ax = plt.subplots(1, 2, figsize=(12, 4))
PartialDependenceDisplay.from_estimator(
    model, X_train,
    features=['age', 'income'],  # Feature indices or names
    kind='average',              # 'average' for PDP, 'individual' for ICE
    ax=ax
)
plt.tight_layout()

# ICE curves (Individual Conditional Expectation) show per-sample variation
fig, ax = plt.subplots(figsize=(8, 4))
PartialDependenceDisplay.from_estimator(
    model, X_train,
    features=['age'],
    kind='both',               # PDP line + ICE curves
    ax=ax
)
PDP vs. ICE vs. ALE — when to use each. PDP shows the average effect and is the most common, but it hides heterogeneous effects (where a feature helps some predictions and hurts others). ICE (Individual Conditional Expectation) plots one line per data point, revealing these heterogeneous relationships — if ICE curves cross or diverge, the feature has interaction effects. ALE is the unbiased replacement for PDP when features are correlated; it should be your default choice. As a rule: start with ALE for global feature effects, overlay ICE when you suspect interactions, and use PDP only when features are known to be independent.

38.4 SHAP Values

SHAP (SHapley Additive exPlanations) is currently the most widely used local explanation method in both research and industry. It is rooted in cooperative game theory: the Shapley value from economics fairly distributes a coalition's payout among its players. In the ML analogy, the "payout" is the model's prediction for a specific input, and the "players" are the input features. The Shapley value $\phi_i$ of feature $i$ is its average marginal contribution across all possible feature coalitions:

$$\phi_i = \sum_{S \subseteq N \setminus \{i\}} \frac{|S|!(|N| - |S| - 1)!}{|N|!} [f(S \cup \{i\}) - f(S)]$$

This formula enumerates every possible subset $S$ of features that excludes $i$, computes the model's prediction with and without feature $i$ (marginal contribution), and takes a weighted average where the weight depends on the coalition size. The key mathematical properties that make Shapley values special are:

  • Efficiency: The Shapley values sum to the difference between the model's prediction and the baseline (expected) prediction: $\sum_i \phi_i = f(x) - E[f(X)]$. This means the explanation is complete — no prediction value is left unaccounted for.
  • Symmetry: If two features contribute identically to every coalition, they receive equal Shapley values.
  • Dummy: If a feature never changes the prediction regardless of which coalition it joins, its Shapley value is exactly zero.
  • Additivity: For ensemble models, the Shapley value of a feature in the ensemble equals the sum of its Shapley values in each component model.

No other attribution method satisfies all four properties simultaneously — this is why SHAP has become the de facto standard for per-prediction explanations in regulated industries.

The computational challenge. Computing exact Shapley values requires evaluating the model on $2^{|N|}$ feature subsets, which is intractable for datasets with more than ~15 features. The SHAP library provides several approximations that make it practical:

Reading a SHAP waterfall plot. The waterfall plot is the most informative single-prediction visualization. It starts from the base value (the model's average prediction across the training set) on the left, and each feature's SHAP value extends the bar to the right (positive contribution, red) or pulls it to the left (negative contribution, blue). The final prediction is where all bars accumulate. Features are ordered by the magnitude of their contribution. For a loan denial, the waterfall might show: base = 0.5 (50% approval rate), income = −0.25, credit_history = −0.15, debt_ratio = −0.10, employment_years = +0.05 → final score = 0.05 (denied). This gives the applicant and the regulator a complete, additive decomposition of the decision.

38.5 LIME (Local Interpretable Model-agnostic Explanations)

LIME takes a fundamentally different approach from SHAP. Instead of computing exact feature attributions, LIME builds a local surrogate model — a simple, interpretable model (typically a sparse linear regression) that approximates the black-box model's behavior in the neighborhood of a specific input. The intuition is: while the global model may be too complex to explain, its behavior near any single point is approximately linear, and a linear model is intrinsically interpretable.

The algorithm works as follows: (1) generate perturbed versions of the input by randomly turning features on/off (for tabular data) or masking image regions (for images), (2) query the black-box model on each perturbation, (3) weight the perturbations by their proximity to the original input (closer perturbations get higher weight), (4) fit a weighted sparse linear model $g$ that minimizes the fidelity loss $\ell(f, g, \pi_x) + \Omega(g)$, where $\ell$ measures how well $g$ approximates $f$ near $x$, and $\Omega$ penalizes complexity (e.g., the number of non-zero coefficients). The resulting linear coefficients are the local feature importances.

    X_train.values,

    mode='classification'

    X_test.iloc[0].values,

LIME vs. SHAP — when to use which. LIME is faster for a single explanation (especially with KernelSHAP as the alternative), but it is less stable — running LIME twice on the same input can produce noticeably different explanations because the perturbation sampling is random. SHAP is deterministic (for TreeSHAP and LinearSHAP) and has stronger theoretical guarantees (efficiency, additivity), but it is computationally more expensive. For regulated industries where explanations must be reproducible and auditable, SHAP is the standard. For rapid prototyping and debugging where speed matters more than rigor, LIME is a pragmatic choice. In practice, most teams use both: SHAP for production explanation APIs and LIME for interactive debugging in notebooks.

38.6 Deep Learning Explanation Methods

Neural networks pose a unique challenge for explainability because their predictions emerge from millions of distributed parameter interactions rather than a small number of human-interpretable features. Specialized methods have been developed that exploit the differentiable structure of neural networks to produce explanations more faithful than generic black-box methods.

38.6.1 Saliency Maps and Gradient-Based Methods

The simplest neural network explanation is the saliency map: compute the gradient of the output (for the predicted class) with respect to each input pixel (or feature), and visualize the gradient magnitude. Intuitively, pixels with large gradients are the ones where a small change would most change the prediction — they are the pixels the model is "most sensitive to." While simple, raw saliency maps have been shown to be unreliable: they can highlight edges and textures that humans find meaningful but the model is not actually using, and they are sensitive to architectural choices unrelated to the model's actual reasoning. Two improved variants address these issues:

  • Integrated Gradients computes the gradient not at a single point but along a straight-line path from a baseline input (e.g., a black image) to the actual input, and integrates the gradients along this path. This satisfies two axioms — sensitivity (if a feature changes and the prediction changes, that feature gets non-zero attribution) and implementation invariance (two functionally equivalent networks get the same attribution) — that raw saliency maps violate. Integrated Gradients is the recommended default for tabular and text neural networks.
  • SmoothGrad averages saliency maps over many copies of the input with small Gaussian noise added, reducing visual noise and producing cleaner, more human-readable visualizations. It is a post-processing step that can be combined with any gradient-based method.
# Integrated Gradients using Captum (PyTorch)

38.6.2 Grad-CAM (Gradient-weighted Class Activation Mapping)

Grad-CAM is the most widely used explanation method for convolutional neural networks (CNNs) in computer vision. It produces a coarse heatmap (the same spatial resolution as the final convolutional feature map) highlighting which regions of the image most influenced the model's classification decision. The algorithm exploits the spatial structure of CNN feature maps: the final convolutional layer produces a set of feature maps, each detecting a different visual pattern at a different spatial location, and Grad-CAM computes a weighted sum of these feature maps where the weights are the average gradients of the target class score with respect to each feature map channel.

$$\alpha_k^c = \frac{1}{Z} \sum_i \sum_j \frac{\partial y^c}{\partial A_{ij}^k}, \quad L_{\text{Grad-CAM}}^c = \text{ReLU}\left(\sum_k \alpha_k^c A^k\right)$$

where $A^k$ is the $k$-th feature map from the final convolutional layer, $y^c$ is the score for class $c$, $\alpha_k^c$ is the importance weight for feature map $k$ for class $c$, and the ReLU ensures only features that have a positive influence on the class of interest are highlighted. The resulting heatmap is upsampled to the input image size and overlaid as a semi-transparent color map. Grad-CAM is class-discriminative — it highlights different regions for "dog" vs. "cat" even in the same image — which raw saliency maps cannot do.

# Grad-CAM using pytorch-grad-cam
from pytorch_grad_cam import GradCAM
from pytorch_grad_cam.utils.image import show_cam_on_image

# Target the final convolutional layer

# Generate heatmap for a specific class
grayscale_cam = cam(input_tensor, targets=None)  # None = predicted class

# Overlay on original image

    original_image_np,
    grayscale_cam[0, :],

)
Grad-CAM vs. saliency maps — which to use? Saliency maps show pixel-level sensitivity (high resolution but noisy, not class-discriminative). Grad-CAM shows region-level importance (lower resolution but clean, class-discriminative, and more human-interpretable). For most trust-building and debugging use cases, Grad-CAM is preferred because it answers the question "which region of the image did the model look at?" rather than "which pixels would change the output the most?" — the former is what clinicians and users actually want to know. For fine-grained analysis (e.g., "is the model looking at the tumor margin or the center?"), combine Grad-CAM with Guided Backprop for higher-resolution Guided Grad-CAM.

38.7 Counterfactual Explanations

Counterfactual explanations answer a fundamentally different question from SHAP or LIME. Instead of "which features contributed to this prediction?", they answer: "what is the smallest change I could make to get a different outcome?" This is often the most actionable form of explanation — a loan applicant doesn't care that their debt ratio contributed −0.3 to the score; they want to know "if my income were $5,000 higher, I would have been approved."

Formally, a counterfactual for input $x$ with prediction $f(x) = y_{\text{unfavourable}}$ is an input $x'$ such that: (1) $f(x') = y_{\text{favorable}}$ (the prediction flips), (2) $d(x, x')$ is minimized (the change is as small as possible), and (3) $x'$ is plausible (it could realistically occur — you cannot change someone's age from 25 to 80). The distance function $d$ is typically a weighted $L_1$ norm over mutable features, and plausibility is enforced by constraining changes to feasible ranges or requiring that $x'$ lies near actual training data points.

# Counterfactual using the DiCE library

Counterfactual explanations have a unique advantage in regulated settings: they provide actionable recourse. The GDPR's "right to explanation" is increasingly interpreted as a right to receive a counterfactual — not just an attribution of the current decision, but guidance on how to achieve a different outcome. However, counterfactuals also carry risks: if multiple valid counterfactuals exist (and they usually do), presenting only one can be misleading (why this path and not another?), and if the model changes, the counterfactual may no longer be valid (recourse must be continuously verified).

38.8 Limitations and Failure Modes of XAI

Critical limitations every ML engineer must understand:
  • Explanation $\neq$ Causation: SHAP values and feature importance measure the model's statistical dependence on a feature, not whether that feature causes the outcome. If a model uses "hospital ID" as a proxy for disease severity (because certain hospitals treat sicker patients), SHAP will correctly report that hospital ID is important — but the explanation is misleading because changing hospitals would not change the patient's health. This is the most common misinterpretation of XAI outputs in practice.
  • Correlated features: When features are highly correlated (e.g., "years of education" and "income"), SHAP and permutation importance can distribute credit arbitrarily between them. One run may assign all importance to education; another may assign it all to income. The total importance is correct, but the per-feature decomposition is unstable. ALE plots and SHAP interaction values can partially address this, but the fundamental ambiguity remains — it is a feature of the data, not a bug in the method.
  • Adversarial manipulation: Research has shown that it is possible to train models that are accurate but produce deliberately misleading explanations — a model can be trained to use a sensitive attribute (like race) while ensuring that SHAP attributes zero importance to it. This "explanation laundering" means that explanations cannot be treated as proof of fairness; they are evidence, not audit results.
  • Rashomon effect: Multiple equally accurate models can produce completely different explanations for the same prediction. The explanation you see depends on the model you happen to deploy, not on any ground truth about the data. This is especially problematic when explanations are used for high-stakes decisions — a different random seed during training could produce a different explanation for the same patient.
  • Medical AI caveat: Heatmaps (Grad-CAM, saliency) are supporting evidence at most. Multiple studies have shown that clinicians tend to over-trust highlighted regions, even when the highlights are deliberately corrupted. Heatmaps must be validated against expert annotations and combined with performance metrics, not presented as standalone proof of correctness.

41.2b Property-Based Testing with Hypothesis

The example-based unit tests above check specific, hand-picked inputs — but hand-picked examples only catch bugs you already thought to test for. Property-based testing flips this around: instead of writing individual examples, you describe a property that should hold for any valid input, and the hypothesis library automatically generates hundreds of randomized (and deliberately adversarial) inputs trying to break that property — empty arrays, single-element arrays, values at numeric extremes, NaNs where allowed, and so on.

Install it alongside your other test dependencies:

pip install hypothesis  # add to requirements-dev.txt alongside pytest, great_expectations

The core idea is simple: you write a property (an invariant that should always hold), decorate your test with @given, and hypothesis supplies the inputs. The library runs each test many times with different randomly generated values, and it specifically tries adversarial edge cases — zero-length inputs, maximum-size inputs, boundary values, NaN, infinity, and so on — that human authors of example-based tests routinely forget. The two examples below illustrate basic usage:

from hypothesis import given, strategies as st
import numpy as np

@given(
    st.lists(
        st.lists(st.floats(min_value=-1e6, max_value=1e6, allow_nan=False), min_size=1, max_size=10),
        min_size=1, max_size=50,
    )
)
def test_scaler_output_has_zero_mean(X):
    """For any valid input matrix, the fitted scaler's output should have ~zero mean per column."""
    X = np.array(X)
    scaler = SklearnCompatibleScaler().fit(X)
    X_scaled = scaler.transform(X)
    assert np.allclose(X_scaled.mean(axis=0), 0, atol=1e-6)

@given(st.lists(st.integers(min_value=0, max_value=1), min_size=1, max_size=1000))
def test_accuracy_is_between_zero_and_one(y_true):
    """Accuracy against a random prediction should always be a valid probability."""
    y_pred = [1 - y for y in y_true]  # deliberately adversarial: always wrong where possible
    acc = compute_accuracy(y_true, y_pred)
    assert 0.0 <= acc <= 1.0

Property-Based Testing for Custom Transformers

Where property-based testing really shines is custom transformers and feature-engineering code — the code you write yourself rather than importing from scikit-learn. These are precisely the components most likely to contain subtle bugs on edge-case inputs: a column of all-identical values, a categorical feature with a single level, a group with zero variance, or a numeric column where every value is zero. Hand-written example tests rarely cover these cases, and they are exactly the cases that cause silent data corruption in production. Below is a worked example for a custom log-transformer with a configurable floor:

from hypothesis import given, settings, strategies as st
import numpy as np
from sklearn.base import BaseEstimator, TransformerMixin

# --- The custom transformer under test ---
class LogPlusConstantTransformer(BaseEstimator, TransformerMixin):
    """log(x + c) where c > 0 ensures finite output even when x == 0."""
    def __init__(self, constant: float = 1.0):
        self.constant = constant

    def fit(self, X, y=None):
        return self

    def transform(self, X):
        X = np.asarray(X, dtype=float)
        return np.log(X + self.constant)

# --- Property-based tests ---
# Strategy: generate 2-D arrays of non-negative floats (common in count features)
nonneg_matrix = st.lists(
    st.lists(
        st.floats(min_value=0.0, max_value=1e8, allow_nan=False, allow_infinity=False),
        min_size=1, max_size=20,    # 1–20 columns
    ),
    min_size=1, max_size=100,      # 1–100 rows
)

@given(X=nonneg_matrix, constant=st.floats(min_value=1e-3, max_value=1e6, allow_nan=False))
def test_log_transform_always_finite(X, constant):
    """log(x + c) must never produce NaN or Inf for non-negative x and positive c."""
    transformer = LogPlusConstantTransformer(constant=constant).fit([])
    result = transformer.transform(X)
    assert np.all(np.isfinite(result)), f"Got non-finite with c={constant}"

@given(X=nonneg_matrix)
def test_log_transform_is_monotone_increasing(X):
    """For each row, if input[i] > input[j] then output[i] > output[j]."""
    transformer = LogPlusConstantTransformer(constant=1.0).fit([])
    result = transformer.transform(X)
    for row_in, row_out in zip(X, result):
        sorted_in = np.argsort(row_in)
        sorted_out = np.argsort(row_out)
        np.testing.assert_array_equal(sorted_in, sorted_out,
            err_msg="Monotonicity violated: log reorder the values")

@given(X=nonneg_matrix)
def test_log_transform_output_non_negative(X):
    """log(x + 1) >= 0 for all x >= 0, because log(1) = 0 and log is increasing."""
    transformer = LogPlusConstantTransformer(constant=1.0).fit([])
    result = transformer.transform(X)
    assert np.all(result >= -1e-12), "log(x+1) should be >= 0 for x >= 0"

These three properties — finiteness, monotonicity, and range constraints — are the bread-and-butter invariants you should write for any mathematical transformer. Notice how each property is simple to state and independent of the implementation details: even if you later swap np.log for a fast approximation, the same tests continue to guard against regressions. Contrast this with example-based tests that hard-code specific expected outputs and break every time the implementation changes.

When a property-based test fails, hypothesis automatically shrinks the failing input down to the smallest example that still reproduces the bug — instead of a 500-row randomly generated array, you get a 2-row minimal failing case, which is dramatically easier to debug. This is especially valuable for custom transformers and feature-engineering code: edge cases like a column of all-identical values, a single row, or an empty group are exactly the inputs that hand-written example tests tend to forget, and exactly the inputs that cause production incidents.

When to add property-based tests: Not every function needs them. Add hypothesis tests when the code (a) accepts a wide range of valid inputs, (b) must satisfy a mathematical invariant, and (c) is complex enough that hand-written examples may miss edge cases. Custom transformers, metric computations, and loss functions are the highest-value targets. For simple glue code or thin wrappers over scikit-learn, example-based pytest tests are usually sufficient and faster to run.

The bug hiding in the code above: the predict endpoint is declared async def, but model.predict_proba(...) is a synchronous, CPU-bound call. FastAPI's async event loop is built around I/O-bound concurrency — while one request is waiting on a network call or disk read, the event loop switches to handle another request. A CPU-bound call like scikit-learn inference (or any non-awaited blocking work) does not yield control back to the loop; it runs to completion and blocks every other request on that worker for its entire duration. For a small, fast model this is easy to miss in local testing; under real traffic with a large model, latency for unrelated requests spikes in lockstep with inference time.

42.1b I/O-Bound vs. CPU-Bound: Fixing the Blocking Call

The fix depends on how expensive inference actually is:

  • Fast, lightweight models (small scikit-learn / XGBoost models, a few milliseconds per call): offload the blocking call to a thread pool with FastAPI's run_in_threadpool, so the event loop stays free while inference runs on a worker thread.
  • Larger or GPU-bound models: don't run inference inside the request/response cycle at all. Push the request onto a message queue (Celery + Redis or RabbitMQ) and let a separate pool of worker processes — sized independently of the web tier, and able to batch requests — pull jobs off the queue and write results back. The API layer stays fast and I/O-bound; the actual computation happens elsewhere and can scale (and be GPU-provisioned) independently.
  • High-throughput GPU serving at scale: use a purpose-built inference server such as NVIDIA's Triton Inference Server, which handles dynamic batching, multi-model serving, and GPU scheduling far better than a hand-rolled FastAPI wrapper ever will.
# Option 1: offload a fast, synchronous call to a thread pool
from fastapi.concurrency import run_in_threadpool

@app.post("/predict", response_model=PredictionResponse)
async def predict(request: PredictionRequest):
    features = [[request.age, request.tenure,
                 request.monthly_charges, request.total_charges]]
    # Runs on a worker thread; the event loop is free to handle other requests meanwhile
    prob = await run_in_threadpool(lambda: model.predict_proba(features)[0][1])
    return PredictionResponse(
        churn_probability=float(prob),
        churn_prediction=prob > 0.5,
        confidence="high" if abs(prob - 0.5) > 0.3 else "medium",
    )

# Option 2: hand off heavier inference to a Celery worker via a message queue
from celery import Celery

celery_app = Celery("inference", broker="redis://localhost:6379/0")

@celery_app.task
def run_inference(features: list) -> float:
    return float(model.predict_proba([features])[0][1])  # runs in a separate worker process

@app.post("/predict-async")
async def predict_async(request: PredictionRequest):
    features = [request.age, request.tenure, request.monthly_charges, request.total_charges]
    task = run_inference.delay(features)   # enqueue and return immediately
    return {"task_id": task.id, "status": "queued"}

44.1 Monitoring Dimensions

Ground-truth delay. Table 44-1 lists "Performance (Accuracy, AUC)" as a monitoring dimension, but in many real systems you simply don't have the true label at prediction time. In fraud detection, you may not know whether a transaction was actually fraudulent for 30–90 days, until a chargeback is filed or an investigation closes. In credit risk, default status may take months to resolve. This means direct performance monitoring is only available on a long lag — if you wait for ground truth before noticing a problem, the model may have been degrading silently for weeks.

44.1b The Ground-Truth Delay Problem

Ground-truth delay is one of the most underappreciated challenges in production ML, and it catches interns off guard because academic settings and Kaggle competitions always provide labels immediately. In the real world, the delay between a prediction and its true label can range from hours (e.g., next-day click confirmation) to months (e.g., credit default, insurance claim resolution), and in some domains like clinical trial outcomes, the lag can be years. During that entire window, you are flying blind on actual model quality — you cannot compute AUC, precision, or recall because you do not know which predictions were correct.

This matters because model degradation does not wait for labels. A fraud model that began silently drifting on day 5 may not show up in AUC metrics until day 60 when chargebacks are finally reconciled — by which point the model has been making poor decisions for eight weeks. The engineering discipline of proxy metric monitoring exists precisely to close this gap: you monitor signals that are available immediately and that correlate with (but do not directly measure) eventual model performance.

44.1c Proxy Metrics for the Ground-Truth Gap

The key insight is that while you cannot observe "was the prediction correct?" you can observe upstream signals that shift when the model's relationship to the data changes. The table below lists practical proxy metrics organized by how quickly they are available:

Interns should internalize this trade-off: the question is never "do we have a perfect real-time performance metric?" but rather "can we detect a problem soon enough to act on it?"

                 score_psi_threshold: float = 0.25, feature_ks_threshold: float = 0.1):
        self.ref_score_mean = reference_scores.mean()
        self.ref_score_std  = reference_scores.std()
        self.ref_features   = reference_features
        self.score_psi_threshold  = score_psi_threshold
        self.feature_ks_threshold = feature_ks_threshold

    def check_prediction_drift(self, current_scores: pd.Series) -> dict:
        """Has the distribution of model outputs shifted? Available immediately."""
        z_shift = abs(current_scores.mean() - self.ref_score_mean) / self.ref_score_std
        psi = self._psi(self.ref_scores if hasattr(self, '_ref_score_hist') else current_scores,
                        current_scores)
        alert = z_shift > 2.0 or psi > self.score_psi_threshold
        return {"metric": "prediction_drift", "z_shift": z_shift, "psi": psi,
                "alert": alert, "available": "immediate"}

    def check_feature_drift(self, current_features: pd.DataFrame) -> list:
        """Has any input feature drifted relative to reference? Available immediately."""
        alerts = []
        for col in self.ref_features.columns:
            if col in current_features.columns:
                stat, p_value = ks_2samp(self.ref_features[col].dropna(),
                                         current_features[col].dropna())
                if stat > self.feature_ks_threshold:
                    alerts.append({"feature": col, "ks_stat": stat, "p_value": p_value})
        return alerts

    def check_manual_review_overturn(self, overrides: pd.Series, total_reviewed: int,
                                      threshold: float = 0.3) -> dict:
        """Are human analysts overturning the model more than usual? 1-7 day lag."""
        overturn_rate = overrides.sum() / max(total_reviewed, 1)
        return {"metric": "overturn_rate", "value": overturn_rate,
                "alert": overturn_rate > threshold, "available": "1-7 days"}

    @staticmethod
    def _psi(expected: pd.Series, actual: pd.Series, bins: int = 10) -> float:
        """Population Stability Index between two distributions."""
        expected_hist = np.histogram(expected, bins=bins, density=True)[0]
        actual_hist   = np.histogram(actual,   bins=bins, density=True)[0]
        expected_hist = np.clip(expected_hist, 1e-4, None)  # avoid log(0)
        actual_hist   = np.clip(actual_hist,   1e-4, None)
        return float(np.sum((actual_hist - expected_hist) * np.log(actual_hist / expected_hist)))

This monitor checks three signals at three different time horizons: prediction-drift and feature-drift are available the instant a batch of predictions is made; the manual-review overturn rate arrives days later but carries a much stronger signal that the model is wrong. None of these prove the model is degrading — a spike in predicted fraud scores could simply mean a holiday shopping surge — but each is a reason to investigate sooner rather than waiting 60 days for chargeback labels.

Practical advice for interns: When you deploy your first model, ask your mentor two questions: (1) "How long until we get ground-truth labels for today's predictions?" and (2) "What proxy metrics can we monitor in the meantime?" If the answer to (2) is "nothing," that is a red flag — it means a silent degradation could go undetected for the entire ground-truth delay window. At minimum, track the distribution of your model's predicted scores and the KS statistic on your most important input features; these cost almost nothing to compute and catch a large fraction of real-world incidents.
